## 作成ルール（重要）

### 1) Markdownの体裁
- 先頭見出しは `# <CATEGORY> (<日本語名>) YAML スキーマ定義` の形式。
- 次に1〜2行でカテゴリ説明。
- `## スキーマ定義` で **表形式**（キー / 型 / 必須 / デフォルト / 説明）を作る。
- 必要に応じて `###` 見出しで列挙値・参照規約・補足を追加。
- 最後に `## YAML 例` を **必ず** 付ける（最小1例、できれば代表パターン2例）。

### 2) 既存設計との整合（今回の反省点の反映）

#### 2-1) 確率の単位は原則 `rate: 0〜100`（Integer）
- 既存の `loot.pool` が `rate: 0〜100` なので、同じDBでは確率表現を原則これに揃える。
- 例: 100=必ず発動、50=50%、0=発動しない。
- 例外的に `0.0〜1.0` が必須な根拠がある場合のみ採用し、理由も明記する。

#### 2-2) “割合/固定”の表現は「概念」に合わせて分離
- `buff` はステータス補正の計算方式なので `modifiers[].type = FLAT/SCALAR`（加算/乗算）を使う。
- `consumable` の回復のように「値の解釈（固定量 or 割合）」が本質の場合は `isPercent` などで表現してよい。
- **同じ `type` に別概念（効果カテゴリと単位切替など）を混ぜない**。

#### 2-3) 参照（ref）のprefix/aliasesを必ず明記
- 参照先DBに合わせて `ref: <prefix>:<id>` の形式で記述。
- 例: `buff` は `buff:`（aliases: `bf`）など。
- スキーマ定義.md側にも `### 参照（ref）` を設け、prefixルールを書く。

#### 2-4) 型・デフォルト・必須の考え方
- “必須”はそのカテゴリが成立する最小構成に必要なものだけを○。
- “デフォルト”は実装/データ運用で自然に決められる値だけを書き、曖昧なら `-`。
- 値の範囲（例: `0〜100`）や単位（tick, 秒, % 等）は説明に必ず含める。

### 3) 出力に含めること（必須）
- キー一覧の表
  - ネスト構造は既存ドキュメントに合わせて `xxx[].yyy[].zzz` の表記で書く（配列/リストが絡む場合）。
- 列挙値があるキーは `### <key>` で取りうる値を列挙。
- `ref` がある場合は `### 参照（ref）` セクション。
- `## YAML 例` セクション。

### 4) 出力に含めないこと（禁止）
- 実装コードの変更提案（このmd作成が目的のため）。
- 想像でのクラス名やパッケージ名の断定（根拠がある場合のみ）。
